# Agent Guide

## 开始说明

先查看项目中的文档，了解当前项目的需求以及开发进度。熟悉项目代码。
中文回复，言简意赅，巧用Emoji。
按需使用 Plan Mode。仅在任务可安全拆分且不阻塞主线程时使用 Subagent（合理安排）。
若编码 Coding，按需使用 skills：`superpowers:*` 和 `karpathy-guidelines`

## 任务流程

1. **记录**：写入 TASK_QUEUE.md，先标记待确认
2. **评估**：文档修改、简单缺陷修复，或小于 100 行且不引入新行为的改动可直接处理；新功能、跨模块改动或方案不明确时先预研
3. **架构**：编码前，做好架构设计和文件夹安排，不要一个文件中有太长的代码，该继承，引用等等的要提前做好。
4. **回归**：根据改动大小决定测试范围
5. **汇报**：汇总结果，列出修改文件

## 测试要求

编码任务优先采用 TDD：先写能失败的 unit/component test，再实现；完成后按改动风险补充必要的 BDD/UI 验证。若 unit/component tests 已足够覆盖，则不强制补 UI 测试。

## 多 Agent 要求

仅在任务可安全拆分、且不阻塞主线程理解与决策时，尽可能并行使用多 Agent；主线程负责问题理解、方案收敛与最终提交。

## 提交要求

检查项目是否有 git。
如果没有 git，建立 git。
有 git，每完成一个小功能或者问题修复之后，进行提交，编写好提交记录，是 bug，还是 feat，还是 improve。
要写中文Log包括：问题或需求描述 或 修复或实现思路 或 复现路径(可选)。
提交完再进行下一个功能的开发或者问题修复。这个过程不需要向我汇报。

## 调研要求

如果项目已经有 DESIGN.md、标准文档、需求文档或现成表格/模板，通常希望严格按这些内容作为实现依据。
